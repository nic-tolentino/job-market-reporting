# 7.1 — Crawler Job Persistence

**Status:** Planned
**Priority:** High — crawler extracts jobs but they are never written to BigQuery, so they never appear on the main site

---

## Problem

The admin crawler (`POST /api/admin/crawler/companies/{id}/crawl`) and the future batch crawler both call the crawler-service and receive a `CrawlResponse` containing:

```json
{
  "crawlMeta": { "totalJobsFound": 7, ... },
  "jobs": [ { "title": "Senior Angular Developer", "location": "Auckland, NZ", ... } ],
  "extractionStats": { ... }
}
```

The backend currently persists the **crawl run audit record** (`crawl_runs`) and updates the **seed health** (`crawler_seeds`), but the `jobs` array is discarded. It is returned to the admin UI caller and then lost. Nothing ever writes to `raw_jobs`.

**Consequence:** Jobs extracted by the crawler never appear on the main website, in tech pages, company profiles, or anywhere else that reads from `raw_jobs`.

---

## Goals

1. After every successful crawl (admin-triggered or future batch), persist extracted jobs to `raw_jobs` using the same Silver Layer infrastructure as the Apify pipeline.
2. Deduplicate against existing records so re-crawling the same company doesn't inflate job counts.
3. Extract technologies from job titles/descriptions using the existing `RawJobDataParser.extractTechnologies()`.
4. Parse locations into the structured `locations: List<String>` format the frontend expects.
5. Keep the mapping logic in a dedicated, testable class — do not bloat `CrawlerAdminController`.

---

## Architecture

```
CrawlerAdminController.triggerCrawl()
  │
  ├─ [existing] crawlRunRepository.append(record)
  ├─ [existing] crawlerSeedRepository.upsert(seed)
  │
  └─ [NEW] CrawlerJobPersistenceService.persist(companyId, normalizedJobs)
              │
              ├─ CrawlerJobMapper.toJobRecords(jobs, companyId)   [NEW]
              │     uses: RawJobDataParser (existing)
              │     uses: IdGenerator (existing)
              │
              ├─ SilverDataMerger.mergeJobs(new, existing)        [existing]
              │
              └─ JobBigQueryRepository.saveJobs(merged)           [existing]
```

### Why not reuse `RawJobDataMapper`?

`RawJobDataMapper` is purpose-built for the Apify `ApifyJobDto` format, which includes platform job IDs, lifecycle grouping (multiple snapshots of the same posting), company logo/description, and benefit arrays from LinkedIn. `NormalizedJob` is simpler — one record per job, already deduped by the crawler. Feeding it through `RawJobDataMapper` would require shimming it into `ApifyJobDto` and running unnecessary lifecycle logic. A dedicated `CrawlerJobMapper` is cleaner and more maintainable.

## New Classes

### 1. `CrawlerJobMapper` — `sync/CrawlerJobMapper.kt`

Maps `List<NormalizedJob>` (from the crawler TypeScript service) to `List<JobRecord>` (BigQuery Silver Layer).

**Key mapping decisions:**

| `NormalizedJob` field | `JobRecord` field | Notes |
|---|---|---|
| `title` | `title` | Direct |
| `companyName` | `companyName` | Direct; `companyId` derived via `IdGenerator.buildCompanyId()` |
| `location` | `locations`, `country`, `city` | Parsed via `RawJobDataParser.parseLocation()` |
| `descriptionText` + `title` | `technologies` | Extracted via `RawJobDataParser.extractTechnologies()` |
| `title` + `seniorityLevel` | `seniorityLevel` | Via `RawJobDataParser.extractSeniority()` |
| `applyUrl` | `applyUrls` | Wrapped in list |
| `platformUrl` | `platformLinks` | Wrapped in list |
| `platformId` | `platformJobIds` | Wrapped in list |
| `salaryMin` / `salaryMax` | `salaryMin` / `salaryMax` | Wrapped as `NormalizedSalary(amount, currency, period="ANNUAL", source="CRAWLER", isGross=null)` |
| `employmentType` | `employmentType` | Direct |
| `workModel` | `workModel` | Direct, fall back to `"On-site"` if null |
| `postedAt` | `postedDate` | Parse ISO date string |
| `source` (always `"Crawler"`) | `source` | `"Crawler"` |
| — | `jobId` | `IdGenerator.buildJobId(companyId, country, title, datePart)` |
| — | `lastSeenAt` | `Instant.now()` |
| — | `urlStatus` | `"UNKNOWN"` (health checker will verify later) |
| — | `benefits` | `emptyList()` |
| — | `jobFunction` | `null` |

**`jobId` stability:** Uses the same `IdGenerator.buildJobId(companyId, country, title, datePart)` formula as the Apify pipeline. This means if Apify has already ingested a "Senior Angular Developer" at Trade Me NZ on the same date, the crawler record will merge with it rather than duplicate it (handled by `SilverDataMerger`).

```kotlin
// Rough sketch — not final
@Service
class CrawlerJobMapper(
    private val parser: RawJobDataParser,
) {
    fun toJobRecords(jobs: List<NormalizedJobDto>, companyId: String): List<JobRecord> {
        return jobs.mapNotNull { job ->
            runCatching { mapOne(job, companyId) }
                .onFailure { log.error("Failed to map crawler job '${job.title}': ${it.message}") }
                .getOrNull()
        }
    }

    private fun mapOne(job: NormalizedJobDto, companyId: String): JobRecord {
        val now = Instant.now()
        val location = job.location ?: ""
        val (city, _, parsedCountry) = parser.parseLocation(location)
        val country = if (parsedCountry != UNKNOWN_COUNTRY) parsedCountry.lowercase()
                      else parser.determineCountry(location).lowercase()

        val technologies = parser.extractTechnologies(
            "${job.title ?: ""} ${job.descriptionText ?: ""}"
        )

        val postedDate = job.postedAt?.let {
            runCatching { LocalDate.parse(it.substring(0, 10)) }.getOrNull()
        }
        val datePart = postedDate?.toString() ?: now.atZone(ZoneId.systemDefault()).toLocalDate().toString()
        val jobId = IdGenerator.buildJobId(companyId, country, job.title ?: "unknown", datePart)

        return JobRecord(
            jobId = jobId,
            platformJobIds = listOfNotNull(job.platformId),
            applyUrls = listOfNotNull(job.applyUrl),
            platformLinks = listOfNotNull(job.platformUrl),
            locations = listOf(city).filter { it.isNotBlank() },
            companyId = companyId,
            companyName = job.companyName ?: "Unknown",
            source = "Crawler",
            country = country,
            city = city,
            stateRegion = "",
            title = job.title ?: "Unknown Title",
            seniorityLevel = parser.extractSeniority(job.title ?: "", job.seniorityLevel),
            technologies = technologies,
            salaryMin = job.salaryMin?.let { NormalizedSalary(it, job.salaryCurrency ?: "NZD", "ANNUAL", "CRAWLER", null) },
            salaryMax = job.salaryMax?.let { NormalizedSalary(it, job.salaryCurrency ?: "NZD", "ANNUAL", "CRAWLER", null) },
            postedDate = postedDate,
            benefits = emptyList(),
            employmentType = job.employmentType,
            workModel = job.workModel ?: "On-site",
            jobFunction = null,
            description = job.descriptionText,
            lastSeenAt = now,
            urlStatus = "UNKNOWN",
        )
    }
}
```

### 3. `CrawlerJobPersistenceService` — `sync/CrawlerJobPersistenceService.kt`

Orchestrates: map → fetch existing → merge → save. Isolated from the controller so it can be called from both the admin endpoint and the future batch crawler.

```kotlin
@Service
class CrawlerJobPersistenceService(
    private val mapper: CrawlerJobMapper,
    private val merger: SilverDataMerger,
    private val jobRepository: JobRepository,
) {
    private val log = LoggerFactory.getLogger(CrawlerJobPersistenceService::class.java)

    fun persist(companyId: String, jobs: List<NormalizedJobDto>): Int {
        if (jobs.isEmpty()) return 0

        val newRecords = mapper.toJobRecords(jobs, companyId)
        if (newRecords.isEmpty()) return 0

        // Fetch existing jobs for this company to enable merge-deduplication
        val existingRecords = try {
            jobRepository.findByCompanyId(companyId)
        } catch (e: Exception) {
            log.warn("Could not fetch existing jobs for $companyId — will insert without merge: ${e.message}")
            emptyList()
        }

        val merged = merger.mergeJobs(newRecords, existingRecords)

        return try {
            jobRepository.saveJobs(merged)
            log.info("Persisted ${merged.size} jobs for $companyId (${newRecords.size} new, ${existingRecords.size} existing)")
            merged.size
        } catch (e: Exception) {
            log.error("Failed to persist jobs for $companyId: ${e.message}", e)
            throw e
        }
    }
}
```

## Changes to Existing Classes

### `CrawlerAdminController.triggerCrawl()`

After the existing `crawlerSeedRepository.upsert(updatedSeed)` block, add:

```kotlin
// Persist extracted jobs to raw_jobs
val rawJobs = try {
    @Suppress("UNCHECKED_CAST")
    objectMapper.convertValue(
        result["jobs"] ?: emptyList<Any>(),
        object : TypeReference<List<NormalizedJobDto>>() {}
    )
} catch (e: Exception) {
    log.warn("Could not deserialize jobs from crawl response: ${e.message}")
    emptyList()
}

if (rawJobs.isNotEmpty()) {
    try {
        val saved = crawlerJobPersistenceService.persist(companyId, rawJobs)
        crawlLogService.log(companyId, "INFO", "Persisted $saved jobs to raw_jobs")
    } catch (e: Exception) {
        log.error("Job persistence failed for $companyId: ${e.message}", e)
        crawlLogService.log(companyId, "WARNING", "Job persistence failed: ${e.message}")
        // Non-fatal — crawl result is still returned to the caller
    }
}
```

## Required `SilverDataMerger` check

`SilverDataMerger.mergeJobs()` currently merges by `jobId`. The crawler `jobId` is computed with the same formula (`companyId.country.titleSlug.date`) as the Apify pipeline, so cross-source deduplication will work automatically when a job has been seen by both pipelines.

Verify `SilverDataMerger.mergeJobRecord()` handles a `source = "Crawler"` vs `source = "LinkedIn"` merge gracefully — it should prefer the record with more populated fields (description, salary) regardless of source, which the current implementation already does by taking non-null values.

---

## Risks & Considerations

| Risk | Mitigation |
|---|---|
| `findByCompanyId()` is slow for companies with many jobs | Add `LIMIT 500` — companies with more than 500 active jobs are edge cases; merge will still work for common jobs |
| `raw_jobs` is append-via-stream (not upsert) — dupes possible | `SilverDataMerger` handles this; same `jobId` = same record when BigQuery is eventually queried with dedup |
| Currency default `"NZD"` hard-coded in `CrawlerJobMapper` | Pass `companyId`'s `hqCountry` from manifest to determine default currency, or leave null and let frontend handle it |
| Tech extraction finds nothing for sparse job descriptions | This is acceptable — technologies list will be empty, job still appears on company page |
| Admin crawls don't set `ingestedAt` metadata (no `ingestion_metadata` Bronze row) | The Analytics "Recent Ingestions" panel reads from `ingestion_metadata`. Crawler jobs can be tracked separately or we add a lightweight entry — out of scope for this plan |

## Implementation Steps

1. **Create `NormalizedJobDto.kt`** — straight Kotlin DTO mirroring the TS type. No logic.

2. **Add `findByCompanyId()` to `JobBigQueryRepository`** and expose it on the `JobRepository` interface.

3. **Create `CrawlerJobMapper.kt`** with unit tests covering:
   - Location parsing (NZ city names → country = `"nz"`)
   - Technology extraction from title + description
   - `jobId` generation matches expected format
   - Null-safety for all optional fields
   - Salary mapping with null currency

4. **Create `CrawlerJobPersistenceService.kt`** with unit tests covering:
   - Empty job list returns 0 without calling repository
   - Merge path when existing jobs exist
   - Insert-only path when `findByCompanyId` throws

5. **Wire into `CrawlerAdminController`** — inject `CrawlerJobPersistenceService`, add the persistence call after seed upsert. Non-fatal (log warning, don't fail the HTTP response).

6. **Deploy and test** with Trade Me (7 jobs, Cornerstone ATS) and Xero — verify jobs appear on `/tech/angular/nz` and company profile pages within a few minutes.

7. **Future: batch crawler** — `CrawlerJobPersistenceService.persist()` will be called from the batch runner once built. No changes needed to the service itself.
